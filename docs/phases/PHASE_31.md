# Phase 31 — 3D Solver Execution

Target release: `v1.24.0` (Tentative)
Status: **Shipped**
Goal: Execute the full 3D meshing and CFD pipeline for a generated wing, including `blockMesh`, `snappyHexMesh`, and `simpleFoam`, and parse the resulting 3D aerodynamic coefficients.

---

## 1. Architectural Analysis

This phase builds directly on the 3D case generation from Phase 30 by adding the execution and result-parsing capabilities.

- **OpenFOAM Runner**: The `aerosynthx.openfoam.runner.run_case` function will be updated. Instead of executing a hardcoded list of 2D solver applications, it will be made more generic. It will inspect the `Allrun` script within the generated case directory and execute the commands listed there. Since our 3D case generation already creates an `Allrun` script with `blockMesh`, `snappyHexMesh`, and `simpleFoam`, this approach allows the Python runner to be agnostic of the specific steps, promoting better separation of concerns.
- **Result Parsing**: A new function, `parse_force_coefficients_3d`, will be created in `aerosynthx.openfoam.runner`. This function will be responsible for:
  1.  Identifying that it's a 3D case.
  2.  Calculating the appropriate reference area for a wing (planform area).
  3.  Parsing the 3D forces file generated by OpenFOAM, which has a different format and location than the 2D `coefficient.dat` file.
- **Workflow Integration**: The `_solve_stage` in the `pipeline.py` will be updated to conditionally call the new `parse_force_coefficients_3d` function if the completed run was for a 3D wing. The existing `SolveResult` dataclass is already capable of storing the final coefficients without modification.

This design enhances the existing OpenFOAM execution logic with minimal changes, cleanly separating the new 3D parsing logic while reusing the core command execution harness.

---

## 2. Goals & Constraints

- **Goal**: Successfully execute the `blockMesh`, `snappyHexMesh`, and `simpleFoam` commands for a 3D wing case.
- **Goal**: Correctly parse the 3D forces and moments from the OpenFOAM output.
- **Goal**: Calculate and store the overall wing Cl, Cd, and Cm.
- **Constraint**: The execution will be triggered by the existing `--execute` flag in the CLI and `execute: true` in the API. No new user-facing parameters will be added.
- **Constraint**: The implementation must be robust to the significantly longer run times of 3D CFD. Timeouts will need to be adjusted.
- **Constraint**: All existing 2D workflows must remain unaffected.

---

## 3. Implementation Plan

1.  **`aerosynthx.openfoam.runner`**:
    *   Modify `run_case` to read and execute commands from the `Allrun` script in the case directory instead of a hardcoded list.
    *   Create a new `parse_force_coefficients_3d` function that calculates the wing planform area and parses the 3D forces file.
    *   Update the main `run_case` function to call the appropriate parser (2D or 3D) based on the presence of a `wing.stl` file in the case.
2.  **`aerosynthx.workflow.pipeline`**:
    *   Increase the default solver timeout in `_solve_stage` to accommodate the longer run times of 3D meshing and simulation.
3.  **Testing**:
    *   Update `tests/openfoam/test_runner.py` with a new test that mocks the execution of a 3D `Allrun` script and provides a "golden" 3D forces file to validate the new parser.
    *   Add a skippable end-to-end integration test that runs a full, real 3D case for a simple wing, if OpenFOAM is available in the test environment.

---

## 4. Acceptance Criteria

- [x] The `run_case` function is updated to execute commands from the `Allrun` script.
- [x] A new `parse_force_coefficients_3d` function is implemented and tested.
- [x] The `_solve_stage` in the pipeline correctly orchestrates a 3D run when `execute` is true for a wing intent.
- [x] The final `RunResult` for a 3D run contains the calculated Cl, Cd, and Cm.
- [x] All existing 2D OpenFOAM and XFOIL tests continue to pass.

---

## 5. Risks & Limitations

- **Execution Time**: 3D CFD is extremely time-consuming. A single run can take hours. The default timeout will need to be very generous, and users will need to be warned about the expected run time. The synchronous API is a major limitation here.
- **Meshing Failures**: `snappyHexMesh` can fail for many reasons. The runner needs to be able to detect and report these failures clearly.
- **Resource Consumption**: 3D CFD requires a significant amount of RAM and CPU. Running these cases on a local machine may not be feasible for all users.

---

## 6. Future Extensibility

- **Asynchronous Execution**: This phase will highlight the need for an asynchronous task queue (like Celery) to handle long-running CFD jobs. This is the most critical next step after this phase.
- **Advanced Result Parsing**: The result parsing can be extended to extract more detailed information, such as pressure distributions over the wing surface.
- **Cluster Execution**: The `CommandRunner` protocol could be implemented with a version that submits jobs to an HPC cluster via a scheduler like Slurm, rather than running them locally.
