# Phase 25 — XFOIL Integration for Fast Aerodynamic Analysis

Target release: `v1.18.0` (Tentative)
Status: **Shipped**
Goal: Integrate XFOIL as an optional, fast-analysis path to provide rapid estimation of airfoil performance (Cl, Cd, Cm) without executing the full OpenFOAM CFD pipeline.

---

## 1. Architectural Analysis

This phase introduces a new analysis capability that runs parallel to the existing OpenFOAM workflow. To maintain strict layering and determinism:

- **New Package**: A new `aerosynthx.xfoil` package will be created. This package will be responsible for all interactions with the XFOIL binary, including input generation, execution, and output parsing. It will be a new "engineering" block at the same layer as `aerosynthx.openfoam`.
- **Workflow Integration**: The `aerosynthx.workflow.pipeline` will be the sole orchestrator. It will be updated to accept an `analysis_mode` flag (`"openfoam"` or `"xfoil"`). Based on this flag, it will dispatch the `DesignIntent` to either the existing `build_case` function or a new `run_xfoil` function. This preserves the "workflow depends on engineering" layering.
- **External Dependency**: XFOIL is an external binary. The `aerosynthx.xfoil` package will interact with it via `subprocess`, treating it as a black box. The core `aerosynthx` package will *not* bundle XFOIL. The installation and `PATH` availability of the `xfoil` executable is a documented prerequisite for using this feature. The system must degrade gracefully if `xfoil` is not found.
- **Data Persistence**: The results of an XFOIL run (Cl, Cd, Cm, etc.) will be stored in a new table in the existing SQLite database, linked to the `RunRow`. This leverages the persistence layer already built in Phase 5.

This approach avoids any changes to the `physics` or `geometry` core and introduces the new functionality at the correct orchestration layer, fully respecting the project's architectural principles.

---

## 2. Goals & Constraints

- **Goal**: Provide Cl, Cd, and Cm coefficients for a given airfoil and flow condition in seconds, not minutes.
- **Goal**: Enable future automated optimization loops that require rapid feedback.
- **Constraint**: The XFOIL integration must be optional. The system must remain fully functional for OpenFOAM-only use cases if the `xfoil` binary is not installed.
- **Constraint**: The core library will not add any new mandatory runtime dependencies.
- **Constraint**: The existing OpenFOAM workflow must remain the default and be entirely unaffected when not using the XFOIL mode.

---

## 3. Implementation Plan

1.  **`aerosynthx.xfoil` Package**:
    *   `xfoil.runner.py`: Contains a `run_xfoil` function that takes an `Airfoil` object and a `FlowCondition`. It will:
        *   Check for the `xfoil` executable on the system `PATH`.
        *   Generate a temporary `.dat` file for the airfoil coordinates.
        *   Generate a script of XFOIL commands (e.g., `LOAD`, `OPER`, `ALFA`, `PACC`, `PWRT`).
        *   Execute `xfoil` using `subprocess.Popen`, piping the command script to its stdin.
        *   Capture stdout/stderr to detect convergence errors.
    *   `xfoil.parser.py`: Contains a function to parse the polar file generated by `PWRT` into a structured `XfoilResult` dataclass.
    *   `xfoil.errors.py`: Defines typed exceptions like `XfoilError`, `XfoilNotFoundError`, `XfoilConvergenceError`.
2.  **Schema & Persistence**:
    *   `aerosynthx.workflow.db.py`: Add a new `XfoilResultRow` SQLAlchemy model to store the output coefficients, linked to a `RunRow`.
    *   `aerosynthx.intent.schemas.py`: Add an optional `xfoil_settings` model to `DesignIntent` to control parameters like number of iterations or viscosity, if needed in the future (for now, use sensible defaults).
3.  **Workflow & API**:
    *   `aerosynthx.workflow.pipeline.py`:
        *   Update `Pipeline.run()` to accept an `analysis_mode: Literal["openfoam", "xfoil"] = "openfoam"`.
        *   Add a new private method `_run_xfoil_stage()` that calls the `xfoil.runner`.
        *   Update the main `run` logic to conditionally execute either the OpenFOAM or XFOIL stage.
    *   `aerosynthx.workflow.cli.py`: Add an `--mode xfoil` flag to the `run` command.
    *   `aerosynthx.api.app.py`: Update the `POST /api/v1/runs` endpoint to accept an `analysis_mode` field in its request body.

## 4. Acceptance Criteria

- [x] A new `aerosynthx.xfoil` package is created with `runner` and `parser` modules.
- [x] The `run_xfoil` function successfully executes the `xfoil` binary with a generated coordinate file and command script.
- [x] The `parser` correctly extracts Cl, Cd, and Cm from the XFOIL polar output file.
- [x] The workflow pipeline accepts an `analysis_mode` and dispatches to the correct backend.
- [x] The CLI (`aerosynthx run --mode xfoil`) and API (`POST /runs` with `{"analysis_mode": "xfoil"}`) successfully trigger an XFOIL run.
- [x] XFOIL results are persisted to the database.
- [x] If the `xfoil` binary is not found, the system raises a clear, typed `XfoilNotFoundError` instead of crashing.
- [x] XFOIL convergence failures are caught and reported as a failed workflow stage.
- [x] All existing OpenFOAM tests pass without modification.
- [x] The new XFOIL functionality is tested via mocked subprocess calls and (optionally, if `xfoil` is installed) end-to-end integration tests.

---

## 5. Testing Strategy

- **Unit Tests**:
    *   The `xfoil.runner` will be tested by mocking `subprocess.Popen`. The tests will assert that the correct command-line arguments and stdin script are generated for a given `DesignIntent`.
    *   The `xfoil.parser` will be tested by feeding it sample golden-file XFOIL outputs and asserting that the parsed dataclass is correct.
- **Integration Tests**:
    *   A new test file, `tests/xfoil/test_xfoil_integration.py`, will contain end-to-end tests.
    *   These tests will be decorated with `@pytest.mark.skipif(shutil.which("xfoil") is None, reason="xfoil binary not in PATH")`.
    *   They will run the full pipeline in `xfoil` mode for a NACA 0012 airfoil and assert that the calculated Cl is approximately correct for a given angle of attack.
- **Workflow/API Tests**: The existing test suites for the CLI and API will be extended to include test cases for the new `analysis_mode` flag/parameter.

---

## 6. Risks & Limitations

- **External Binary Management**: The biggest risk is the reliance on a user-installed `xfoil` binary. The setup and documentation must be exceptionally clear about this requirement.
- **Platform Compatibility**: XFOIL must be compiled for the target platform (Linux, macOS, Windows). Our test strategy can only realistically cover the CI platform (likely Linux).
- **Parsing Fragility**: The XFOIL polar file format is text-based and could change. The parser must be written defensively.
- **Scope**: This phase will only support a single-point analysis (one angle of attack). Full polar sweeps (iterating over multiple AoAs) are a natural extension but are out of scope for this initial integration.

---

## 7. Future Extensibility

- **Polar Sweeps**: The `run_xfoil` function can be extended to accept a range of angles of attack and generate a full polar curve.
- **Viscous vs. Inviscid**: The XFOIL command script can be modified to run in different modes (e.g., inviscid).
- **Optimization Loops**: With a fast analysis path in place, a future `aerosynthx.optimizer` package could be built on top of the workflow, iterating through `DesignIntent` variations to optimize for a target Cl/Cd ratio.
